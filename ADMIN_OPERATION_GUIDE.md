# 管理员操作手册（运维+业务）

适用对象：管理员账号（`admin`）  
目标：规范执行用户管理、排班管理、备份恢复和高风险操作

> 文档导航：[`README.md`](./README.md) | [`USAGE_GUIDE.md`](./USAGE_GUIDE.md) | [`DOCTOR_QUICK_GUIDE.md`](./DOCTOR_QUICK_GUIDE.md)

---

## 1. 管理员职责范围

管理员可执行全部后台功能：

- 用户管理（增删改医生账号）
- 手术管理（跨医生查看、编辑、删除）
- 手术状态管理（已安排/已完成/已取消）
- 数据库管理（统计、备份、恢复、清空）

---

## 2. 日常工作清单（建议）

建议每天执行：

1. 检查当日与次日排班冲突
2. 检查异常状态手术（取消/改期）
3. 备份数据库（至少每日 1 次）
4. 抽查账号权限和无效账号

---

## 3. 用户管理

路径：管理员首页 -> 用户管理

## 3.1 新增医生

填写：

- 用户名（唯一）
- 密码
- 姓名
- 角色（doctor）

规则：

- 用户名重复会失败
- 建议使用规范命名（如拼音/工号）

## 3.2 修改用户

可修改：

- 用户名
- 姓名
- 角色
- 密码

注意：

- 修改角色会影响权限边界
- 建议记录改动原因

## 3.3 删除用户

限制：

- 不能删除当前登录管理员自己

建议流程：

1. 先确认账号不再使用
2. 再执行删除
3. 记录操作时间和原因

---

## 4. 手术管理（管理员视角）

## 4.1 新增手术

必填：

- 医生
- 病人姓名
- 手术项目
- 日期
- 手术室
- 开始/结束时间

可选：

- 状态
- 备注

系统自动校验：

- 同手术室同日期时间冲突

## 4.2 编辑手术

管理员可跨医生编辑：

- 日期、时间、手术室、患者、手术项目、状态、备注、归属医生

变更建议：

- 对关键字段变更（时间/手术室/医生）做内部登记

## 4.3 删除手术

删除后数据不可直接恢复，建议先做备份再执行批量删除。

## 5. 数据库管理（重点）

路径：管理员首页 -> 数据库管理

## 5.1 统计

可查看：

- 用户总数
- 手术总数
- 已完成手术数
- 数据库文件大小

用途：

- 快速判断数据规模和健康状态

## 5.2 备份

提供两种备份导出：

- 普通备份：不含密码哈希（用于业务迁移/归档）
- 完整备份：含密码哈希（用于完整恢复）

建议：

- 每天至少备份一次
- 重大改动前额外备份

## 5.3 恢复（restore）

恢复模式：

- `merge`：合并导入（推荐常规使用）
  - 已存在记录会跳过
  - 冲突记录会跳过并提示
- `replace`：替换导入（高风险）
  - 会先清空现有数据再导入

恢复前强制建议：

1. 先备份当前数据库
2. 在低峰时段执行
3. 恢复后抽查关键日期与关键医生排班

## 5.4 清空数据（clear）

这是高危操作，仅保留 admin 账号。  
请务必确保：

- 已完成最新备份
- 已得到业务负责人确认
- 在低峰时段执行

---

## 6. 权限与合规建议

- 生产环境必须更换默认管理员密码
- `JWT_SECRET` 必须使用高强度随机值
- 不在群聊传播管理员账号密码
- 最小权限原则：医生只发 doctor 账号
- 定期清理离职/停用账号

---

## 7. 故障排查

## 7.1 启动失败：sqlite3 架构不匹配

执行：

```bash
npm run rebuild-sqlite3
```

## 7.2 端口冲突（3000/3001）

现象：`EADDRINUSE`  
处理：结束占用进程或修改端口后重启

## 7.3 登录后大量 401/403

检查：

- Token 是否过期
- `JWT_SECRET` 是否被修改
- 用户角色是否被错误变更

## 7.4 数据异常

优先排查：

1. 是否刚执行恢复/清空
2. 是否筛选日期有误
3. 是否权限导致视图受限

---

## 8. 应急处理建议

建议保留一个简单应急流程：

1. 暂停高风险操作（恢复/清空）
2. 导出当前备份
3. 确认最近一次可用备份
4. 在测试环境验证恢复
5. 再进行生产恢复
